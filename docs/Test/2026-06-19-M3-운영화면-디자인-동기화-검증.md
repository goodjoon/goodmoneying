## RED 확인

TDD(Test-Driven Development) 순서를 지키기 위해 구현 전에 다음 실패를 확인했다.

- API 테스트: `DashboardTotals`, `MarketListRow`, `InstrumentDetailResponse`에 디자인용 View Model 필드가 없어 실패
- UI 테스트: 기존 화면이 분리 티커, KST 단일 시간 라벨, TradingView 차트 레이블, 현재가 게이지 요구를 만족하지 못해 실패
- E2E(End-to-End): 기존 선택자가 새 디자인 구조와 맞지 않아 상세 모달과 수집 대상 설정 흐름을 새 사용자 흐름 기준으로 재작성

## 수동 검증

자동 E2E에서 다음 사용자 흐름을 확인했다.

- 첫 화면에서 다크 테마(Dark Theme), 좌측 내비게이션(Navigation), 운영 KPI(Key Performance Indicator), 운영 헬스 패널이 표시됨
- 코인별 수집 상태에서 `BTC / KRW`처럼 기준화폐와 코인명이 분리되어 표시됨
- 시간 관련 화면에서 KST 라벨만 시간 옆에 표시되고 UTC(Coordinated Universal Time)는 별도 배지로 노출되지 않음
- 수집 대상 설정에서 50개 선택 상태, 체크박스(Checkbox) 변경, 저장 버튼 동작이 유지됨
- 시장 리스트에서 상세 대화상자를 열면 TradingView 캔들 차트와 현재가 게이지가 표시됨
- 데스크톱(1440x1000)과 모바일(390x844) 뷰포트(Viewport)에서 대시보드와 상세 화면 스크린샷을 생성함
- 상세 화면 캔버스(Canvas)는 데스크톱과 모바일 모두 nonBlank 픽셀이 1000개를 초과해 빈 차트가 아님을 확인함

## 미검증 항목

- TradingView 호스팅 위젯과 MEXC 외부 차트 데이터 직접 연결은 이번 범위가 아니다. 공식 TradingView Lightweight Charts 라이브러리(Library)를 사용해 내부 API 캔들 데이터를 렌더링했다.
- 실제 장기 운영 데이터에서 저장량 추정치와 실패율이 충분히 의미 있는 분포를 갖는지는 운영 수집량이 누적된 뒤 별도 확인이 필요하다.
- Codex 명령 실행 환경에서는 `./dev.sh app restart all`이 만든 `nohup` 자식 프로세스가 장기 유지되지 않았다. 실제 반영은 `launchctl submit`으로 수행했다.
- `StarletteDeprecationWarning`은 FastAPI TestClient 의존성 경고이며 기능 실패는 아니다.
