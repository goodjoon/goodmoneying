## 수동 검증

- 운영 화면이 첫 화면에서 운영 상태 대시보드를 표시하는지 확인했다.
- Playwright E2E가 API webServer를 실제로 띄우고 API source 표시, 수집 대상 변경 저장, 백필 계획 생성/승인/pause, 시장 리스트, 코인 상세 화면을 검증했다.
- Playwright 스크린샷으로 데스크톱과 모바일 폭에서 주요 UI가 표시되는지 확인했다.
- `review` 스킬 기준 Standards와 Spec 리뷰를 병렬 subagent로 실행했고 P1/P2 발견 사항을 반영했다.
- 최종 Docker/PostgreSQL/E2E 안정화 변경은 subagent 사용량 제한으로 직접 Standards/Spec 리뷰를 수행했고, live 프로필 선택 누락과 E2E 상태 의존성을 수정했다.
- `qa` 관점에서 MVP 사용자 시나리오인 운영 상태 확인, 수집 대상 조정, 백필 제어, 시장 리스트 탐색, 코인 상세 확인을 E2E 경로로 점검했다.

## 환경 특이사항

- `docker` CLI(Command Line Interface)는 현재 로컬 환경에서 없었다.
- 호스트 `psql` 명령도 현재 로컬 환경에서 없었다.
- 대신 현재 로컬 환경의 Podman VM과 standalone `docker-compose` v5.1.1을 사용해 실제 PostgreSQL 18 서버, API, worker, web 컨테이너를 빌드/기동하고 검증했다.
- PostgreSQL schema 적용은 컨테이너 내부 `psql`과 API/worker 동작으로 확인했다.

## 미검증 항목

- 실제 업비트 API 호출은 기본 자동화 테스트에서 제외했다. worker가 `GOODMONEYING_LIVE_UPBIT=1`일 때 live 클라이언트(Live Client)를 선택하는 경로는 검증했고, 외부 API 호출 자체는 live 프로필(profile)로 별도 실행한다.

## 결론

M1 업비트 수집 운영 MVP는 자동화 테스트, 브라우저 E2E(End-to-End), standalone `docker-compose` 기반 컨테이너 실행, PostgreSQL 18 schema 적용과 데이터 저장 검증을 통과했다. 실제 업비트 API 호출만 의도적으로 기본 검증에서 제외하고 live 프로필(profile)로 분리했다.
