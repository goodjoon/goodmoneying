## 검증 대상

- 백필 수집 워커(Backfill Collection Worker) batch 크기 기본값 3000과 외부 설정 `GOODMONEYING_BACKFILL_BATCH_SIZE`
- 이미 저장된 캔들 분(minute)을 제외한 결측 구간(Missing Range) 요청
- fetch 성공 후 heartbeat callback 실행
- DB batch upsert 성공 후 `rows_written_count`, `last_completed_at` 갱신
- pause/stop/resume/safe restart 회귀
- 실패(failed) 백필 작업 재개 버튼과 `resume` 제어
- 실패 작업 재개 시 이미 저장된 데이터 제외 후 결측 구간만 요청
- worker 로그 레벨 설정 `GOODMONEYING_LOG_LEVEL`
- worker 디버그 로그의 job/target/range/fetch/upsert 경계
- DB/API 계약 shape 유지
- 운영 화면 E2E(End-to-End) 회귀

## 수동 확인

- 별도 수동 브라우저 확인은 수행하지 않았다. Playwright E2E가 로컬 API 서버와 웹 서버를 띄워 M1 운영 화면 주요 시나리오를 검증했다.

## 남은 리스크

- 실제 업비트 API 장시간 백필의 처리량과 heartbeat 간격은 운영 데이터량에 따라 추가 관측이 필요하다.
- 백필 동시성(Concurrency) 확대, 메시지 큐(Message Queue), 분산 rate limiter는 이번 범위가 아니며 M3.5 결정 항목으로 남긴다.
