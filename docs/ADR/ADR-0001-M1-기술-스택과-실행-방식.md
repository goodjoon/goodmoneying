# ADR-0001-M1-기술-스택과-실행-방식

Status: Accepted
Date: 2026-06-17

## 맥락

M1 업비트 수집 운영 MVP(Minimum Viable Product)는 수집 워커(Collection Worker), 운영 서버(Operations Server), 운영 화면, DB를 모두 포함하는 제품 세로 절편이다. 데이터 수집과 후속 분석/대규모 언어 모델(LLM, Large Language Model)/백테스트(Backtest) 확장을 고려하면 Python 생태계를 백엔드 중심으로 두는 것이 유리하고, 운영 화면은 대시보드와 차트 UX(User Experience)가 필요하다.

## 결정

M1 기술 스택은 Python, FastAPI, React, PostgreSQL, Docker Compose로 한다. 저장소 구조는 `apps/api`, `apps/worker`, `apps/web`, `packages/shared`로 나누고, M1 실행은 `postgres`, `api`, `worker`, `web` 서비스를 Docker Compose로 실행한다.

## 대안

| 대안 | 장점 | 단점 |
|---|---|---|
| Python + FastAPI + React + Docker Compose | 수집/분석 생태계, OpenAPI 계약, 운영 화면 확장성, 로컬/단일 서버 실행 재현성 | 초기 설정이 Streamlit보다 많음 |
| Python + Streamlit | 매우 빠른 화면 작성 | 후속 설정 화면, 차트, E2E(End-to-End) 테스트, 장기 UI 구조가 약함 |
| Next.js 단일 앱 + Node.js 수집기 | 웹 앱 통합이 단순함 | 데이터 수집, 분석, LLM, 백테스트 확장에서 Python 생태계 이점을 버림 |

## 결과

- 백엔드 API 계약은 `docs/contracts/api/openapi.yaml`로 관리한다.
- DB 계약은 `docs/contracts/db/schema.sql`로 관리한다.
- 프론트엔드(Frontend)는 React Query로 서버 상태를 관리하고, 화면 상태는 로컬 상태로 관리한다.
- M1 운영 화면은 HTTP 폴링(Polling)을 사용한다.

## 후속 작업

- 후속 실시간성 조회 화면이 필요해지면 SSE(Server-Sent Events)와 WebSocket 중 무엇을 쓸지 다시 결정한다.
- Docker Compose 이후 운영 배포 방식은 제품 사용 패턴을 본 뒤 별도 ADR로 결정한다.
