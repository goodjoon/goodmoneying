# ADR-0003-M1-단일-워커와-M3-5-다중화-게이트

Status: Accepted
Date: 2026-06-17

## 맥락

M1의 활성 수집 대상(Active Collection Target)은 업비트(Upbit) KRW 마켓 50개이며, 초기에는 단일 수집 워커(Collection Worker)로 충분하다. 하지만 후속 국내 주식, 미국 주식, 뉴스/공시/리포트, LLM 신호, 전략, 봇이 붙으면 수평 확장(Horizontal Scaling)과 고가용성(High Availability)이 필요해진다.

## 결정

M1은 단일 수집 워커와 단일 운영 서버(Operations Server)로 시작한다. 중복 저장은 DB 유니크 제약(Unique Constraint)과 upsert로 방어한다. 수평 확장과 고가용성 고도화는 M3 이후, M4 국내 주식 확장 전에 필수 게이트(Gate)로 수행한다.

## 대안

| 대안 | 장점 | 단점 |
|---|---|---|
| M1 단일 워커, M3.5 다중화 게이트 | M1 복잡도는 낮추고 시장 확장 전 구조를 강화함 | M1에서는 워커 장애 시 자동 failover가 없음 |
| M1부터 다중 워커와 메시지 큐(Message Queue) | 확장성과 장애 대응이 빠름 | 아직 병목과 데이터량을 모르는 상태에서 과설계 위험이 큼 |
| M5 이후 다중화 | 초기 구현 속도가 빠름 | 국내/미국 주식까지 붙은 뒤 구조 변경 비용이 커짐 |

## 결과

- M1 백필 제어는 DB 상태 폴링(Polling)으로 구현한다.
- 정식 버전에서는 메시지 큐를 사용해 작업 분배와 제어 이벤트를 처리한다.
- 메시지 큐 기술은 지금 고정하지 않고 M3.5에서 반드시 결정한다.
- M1 rate limit은 워커 내부 전역 rate limiter로 관리하고, 다중 워커 전환 시 분산 rate limiter를 다시 결정한다.

## 후속 작업

- M3.5에서 Redis Streams, RabbitMQ, Kafka, NATS 등의 후보를 비교한다.
- 작업 큐 중심인지 이벤트 로그 중심인지, 재처리와 순서 보장이 얼마나 필요한지, 다중 워커 제어 이벤트를 같은 큐로 처리할지 결정한다.
