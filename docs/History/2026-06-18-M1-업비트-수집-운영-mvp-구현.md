## 영향 계약

- 기존 `docs/contracts/db/schema.sql`와 `docs/contracts/api/openapi.yaml`를 구현 기준으로 사용했다.
- API route path는 OpenAPI 계약의 `{instrumentId}`, `{jobId}` 이름과 일치하도록 구현했다.
- PostgreSQL 런타임은 `GOODMONEYING_DATABASE_URL`이 PostgreSQL URL일 때 `docs/contracts/db/schema.sql`을 기준으로 빈 DB에 schema를 적용한다.
- PostgreSQL 18 컨테이너 볼륨은 `/var/lib/postgresql`에 마운트하도록 수정했고, Compose는 `postgres` 헬스체크(Health Check) 이후 API/worker를 시작한다.
- Playwright E2E는 기존 DB 상태에 의존하지 않도록 시작 전 수집 대상 기준 상태를 API로 재설정하고, 백필 작업은 방금 생성한 작업 ID 기준으로 검증한다.
- worker는 기본 fixture 클라이언트(Fixture Client)를 사용하고, `GOODMONEYING_LIVE_UPBIT=1`일 때 live 업비트 클라이언트(Live Upbit Client)를 선택한다.
- 최종 Docker/PostgreSQL/E2E 안정화 변경은 subagent 사용량 제한으로 직접 Standards/Spec 리뷰를 수행했다.

## 리스크

- 현재 로컬 환경에는 `docker` CLI와 호스트 `psql` 명령이 없다. 대신 standalone `docker-compose`를 Podman 소켓(socket)에 연결해 실제 PostgreSQL 서버와 Compose 실행을 검증했다.
- 기본 테스트는 업비트 fixture 기반이다. 실제 업비트 API drift는 `GOODMONEYING_LIVE_UPBIT=1` 프로필로 별도 확인해야 하며, 해당 프로필 선택 경로는 자동화 테스트로 확인했다.
- M1의 데이터 완전성 검사는 coverage와 incident 기반의 최소 구현이다. 결측 구간(Missing Range) 복구 워크플로우 고도화는 후속 개선 여지가 있다.
