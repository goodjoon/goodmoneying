## 리스크

- 계약은 구현 전 초안이므로 실제 업비트 API 응답과 구현 중 정합성 검증이 필요하다.
- M1은 단일 수집 워커 구조라 worker 장애 시 자동 장애 조치(Failover)는 제공하지 않는다.
- M1 삭제 없음 정책은 저장량 증가를 유발하므로 M3 또는 M3.5에서 보존 정책을 반드시 결정해야 한다.
- OpenAPI 계약은 구현 중 FastAPI generated OpenAPI와 drift가 생기지 않도록 검증이 필요하다.

## 후속 작업

- `apps/api`, `apps/worker`, `apps/web`, `packages/shared` 구조를 생성한다.
- `docs/contracts/db/schema.sql`을 실제 PostgreSQL에 적용하는 계약 테스트를 작성한다.
- `docs/contracts/api/openapi.yaml` 기준으로 FastAPI 응답 계약 테스트를 작성한다.
- 업비트 API mock/fixture 테스트와 별도 `live` 테스트 프로필(profile)을 구현한다.
