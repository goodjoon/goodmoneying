## 변경 요약

- 백필 수집 워커(Backfill Collection Worker)의 batch 저장 기준을 문서화하고 구현했다.
- 이미 저장된 캔들 구간은 업비트에 다시 요청하지 않고 없는 결측 구간(Missing Range)만 요청하는 요구사항을 제품 기준에 추가했다.
- fetch 성공 시 heartbeat를 갱신하고, `rows_written`과 `last_completed_at`은 DB batch upsert 완료 후 갱신하는 결정을 ADR(Architecture Decision Record)로 기록했다.
- DB batch 크기는 기본 최대 3000개이며 `GOODMONEYING_BACKFILL_BATCH_SIZE` 외부 설정으로 변경할 수 있게 했다.
- `UpbitClient`에 page iterator 계약을 추가하고, 백필 워커가 page를 batch로 누적해 upsert하도록 변경했다.
- 빈 fetch 결과는 DB batch upsert가 아니므로 `last_completed_at`을 목표 종료 시각으로 앞당기지 않게 했다.
- 실패(failed)한 백필 작업은 Backfill 관리 화면에서 재개(Resume)할 수 있게 했다.
- 실패 작업 재개 시 저장 상태를 다시 계산해 이미 들어간 데이터는 업비트에 요청하지 않고 없는 결측 구간만 요청한다.
- 워커 로그 레벨을 `GOODMONEYING_LOG_LEVEL`로 설정할 수 있게 하고, 백필 job/target/range/fetch/upsert 경계를 로그로 남긴다.

## 리스크

- 실제 업비트 장시간 백필에서 fetch page 단위 heartbeat가 충분한지는 운영 데이터로 추가 관측해야 한다.
- pause/stop 제어가 batch 경계에서 동작하도록 테스트했지만, 다중 워커 환경은 아직 범위 밖이다.
- `DEBUG` 로그는 운영 장애 분석에는 유용하지만 평시에는 로그량이 증가할 수 있으므로 기본은 `INFO`로 유지한다.

## 후속 작업

- 운영에서 `GOODMONEYING_BACKFILL_BATCH_SIZE`를 조정할 때 heartbeat 간격, DB upsert 시간, API quota 사용량을 함께 관측한다.
- 운영 장애 분석 시 `GOODMONEYING_LOG_LEVEL=DEBUG`로 올려 백필 job, target, fetch, DB batch upsert 경계를 확인한다.
- M3.5에서 백필 동시성, 메시지 큐(Message Queue), 분산 rate limiter를 별도 결정한다.
