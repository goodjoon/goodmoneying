## 설계 메모

### 현재까지 합의한 결정

- M1 핵심 경계는 업비트 수집 파이프라인(Upbit Collection Pipeline)으로 부른다.
- 국내 주식, 미국 주식, LLM, 전략, 봇은 후속 모듈로 구현하지만, 아키텍처에는 목표 모듈 지도와 의존 방향을 선반영한다.
- 아키텍처 문서는 목표 아키텍처 골격과 M1 상세 설계를 함께 담는다. 후속 모듈의 상세 계약은 지금 확정하지 않는다.
- 런타임(Runtime)은 수집 워커(Collection Worker)와 운영 서버(Operations Server)를 분리한다.
- M1 저장소(Database)는 PostgreSQL 단일 저장소로 시작한다.
- 캔들(Candle)과 현재가 스냅샷(Ticker Snapshot)은 별도 사실(Fact)로 저장한다.
- 저장하는 원천 캔들(Source Candle)은 1분 캔들과 일봉이다.
- 3분, 5분, 10분, 15분, 30분, 60분, 240분, 주봉, 월봉은 파생 캔들(Derived Candle)로 제공한다.
- 백필(Backfill)과 증분 수집(Incremental Collection)은 별도 워크플로우로 분리한다.
- 후보 유니버스(Candidate Universe)는 자동 갱신하고, 활성 수집 대상(Active Collection Target)은 사용자가 확정한 50개 세트로 유지한다.
- 활성 수집 대상에서 빠진 코인은 비활성 수집 대상(Inactive Collection Target)으로 두고 과거 데이터는 보존한다.
- 현재가 스냅샷, 호가 요약(Orderbook Summary), 1분 원천 캔들, 일봉, 후보 유니버스는 데이터 성격별로 수집 주기를 분리한다.
- 수집 실패는 수집 실행(Collection Run)과 대상별 수집 결과(Target Collection Result)로 분리해 기록한다.
- 재시도(Retry)는 짧은 즉시 재시도, 실패 기록, 후속 백필 복구로 처리한다.
- 현재가 스냅샷과 호가 요약은 M1에서 1분 단위 원본을 보존하며, 삭제/다운샘플링(Downsampling)은 후속으로 둔다.
- 호가 요약은 상위 10호가 기준으로 저장한다.
- 데이터 품질(Data Quality)은 수집 시도 품질(Collection Attempt Quality)과 데이터 완전성 품질(Data Completeness Quality)로 분리한다.
- 운영 대시보드의 정상/주의/장애 상태는 운영 서버가 계산하고, DB에는 계산 가능한 사실을 저장한다.
- 운영 서버 API는 화면 단위 조회 API와 설정 변경 API로 구성한다.
- 기술 스택은 Python, FastAPI, React, PostgreSQL, OpenAPI로 확정한다.
- 저장소 구조는 `apps/api`, `apps/worker`, `apps/web`, `packages/shared`로 나눈다.
- 실행/배포 단위는 Docker Compose 기반 로컬/단일 서버 실행으로 시작한다.
- 설정(Configuration)은 환경 변수(Environment Variable)와 DB 설정 테이블을 병행한다.
- M1은 단일 수집 워커만 지원하되, DB 유니크 제약(Unique Constraint)으로 중복 저장을 방지한다.
- 수평 확장(Horizontal Scaling)과 고가용성(High Availability) 고도화는 M3 이후, M4 국내 주식 확장 전에 필수 게이트(Gate)로 둔다.
- M1 관측성(Observability)은 구조화 로그(Structured Log), 헬스체크(Health Check), 기본 메트릭(Metric)을 포함한다.
- 수집 진행률(Collection Coverage)은 수집 대상별 데이터 유형이 목표 수집 범위 대비 어느 시점 또는 어느 비율까지 수집됐는지 표시한다.
- 수집 진행률 개념은 향후 국내 주식과 미국 주식 데이터 수집에도 동일하게 적용한다.
- 목표 수집 범위는 데이터 유형별로 다르게 두고, N 값은 관리 화면에서 설정한다.
- 수집 범위 설정은 전역 기본값과 데이터 유형별 오버라이드(Override)를 지원한다. M1에서는 대상별 오버라이드는 제외한다.
- 백필은 자동 계획 생성 후 사용자가 실행 승인한다.
- 백필 작업은 대기, 실행 중, 일시정지(Pause), 중지(Stop), 완료, 실패 상태를 가진다.
- 백필 작업은 일시정지, 중지, 이어서하기(Resume), 처음부터 재시작(Restart)을 지원한다.
- M1의 처음부터 재시작은 안전 재시작(Safe Restart), 즉 기존 데이터 유지와 전체 범위 재검사만 구현한다.
- 삭제 후 재수집(Destructive Rebuild)은 M1 이후에 구현해야 하며, 삭제 전 확인, 감사(Audit) 로그, 삭제 범위 검증, 복구 또는 재실행 정책이 필요하다.
- M1 백필 작업 제어는 DB 상태 폴링(Polling)으로 구현한다.
- 정식 버전에서는 메시지 큐(Message Queue)를 사용해 작업 분배와 제어 이벤트를 처리한다.
- 메시지 큐 기술은 지금 확정하지 않고, M3.5 수평 확장/고가용성 고도화 시점에 반드시 결정 질문으로 남긴다.
- M1 검증은 계약 테스트, 통합 테스트, 브라우저 E2E(End-to-End) 테스트를 포함한다.
- 기본 자동화 테스트는 mock/fixture 기반으로 실행한다.
- 실제 업비트 API를 부분 호출하는 자동화 테스트는 별도 `live` 테스트 프로필(profile)로 분리한다.
- 시간 기준은 저장 시각(Storage Time)을 UTC로 통일하고, 업비트 KRW 마켓 화면의 표시 시각(Display Time)은 KST(Korea Standard Time)를 기본으로 한다.
- 업비트 마켓은 거래 상품(Instrument)으로 모델링하며, `market_code`, `exchange`, `quote_currency`, `base_asset`, `display_name`을 가진다.
- 후보 유니버스(Candidate Universe)는 업비트 현재가 스냅샷(Ticker Snapshot)의 24시간 누적 거래대금 기준 상위 100개로 산정한다.
- 활성 수집 대상(Active Collection Target)이 후보 유니버스에서 이탈해도 자동 제거하지 않고, 후보 유니버스 이탈 상태로 표시한다.
- 후보 유니버스 스냅샷과 활성 수집 대상 변경 이력은 M1부터 보존한다.
- M1 API 인증(Authentication)은 로컬 신뢰 네트워크 전제와 쓰기 API 단순 운영 토큰으로 구성한다.
- M1부터 활성 수집 대상 저장, 수집 범위 설정 변경, 백필 제어 같은 쓰기 작업에 감사 로그(Audit Log)를 남긴다.
- M1 알림(Notification)은 알림 이벤트(Notification Event) 저장과 대시보드 표시까지만 포함하고, 외부 알림 발송은 MVP 이후 필수 결정 항목으로 남긴다.
- 프론트엔드(Frontend) 서버 상태는 React Query로 관리하고, 화면 상태는 로컬 상태로 관리한다.
- 운영 화면 자동 갱신 주기는 화면별로 분리한다.
- M1 수집/운영 화면은 HTTP 폴링(Polling)을 사용하고, 후속 실시간성 화면에서는 SSE(Server-Sent Events) 또는 WebSocket 선택을 다시 질문한다.
- M1 API는 화면 단위 View Model API와 원천 리소스 API를 모두 제공한다.
- 원천 리소스 API는 외부 공개 API가 아닌 내부 안정 계약(Internal Stable Contract)으로 OpenAPI에 포함한다.
- DB 스키마(schema)는 원천 사실 테이블 중심으로 설계하고, 화면용 집계/상태 테이블은 M1에서 최소화한다.
- 원천 캔들(Source Candle)의 유니크 키(Unique Key)는 `(instrument_id, source, candle_unit, candle_start_at)`로 둔다.
- 현재가 스냅샷(Ticker Snapshot)과 호가 요약(Orderbook Summary)의 유니크 키는 `(instrument_id, source, bucket_at)`로 둔다.
- 현재가 스냅샷과 호가 요약에서 같은 수집 버킷 시간(Collection Bucket Time)에 재수집이 발생하면 upsert로 최신 수집 결과가 대표 행을 덮어쓴다.
- 현재가 스냅샷과 호가 요약의 대표 행 갱신 기준은 더 늦은 `collected_at`을 가진 성공 수집 결과다.
- 실패 수집 결과는 현재가 스냅샷 또는 호가 요약 대표 행을 덮어쓰지 않고 대상별 수집 결과(Target Collection Result)에만 기록한다.
- 수집 진행률(Collection Coverage)은 원천 상태를 DB에 저장하고, 진행률 비율과 화면용 상태는 운영 서버(Operations Server)가 계산한다.
- 결측은 결측 구간(Missing Range)으로 명시 저장한다.
- 결측 구간은 데이터 완전성 검사(Data Completeness Check)가 생성하거나 해결한다.
- M1에서는 데이터 완전성 검사 작업을 수집 워커(Collection Worker) 안의 별도 작업으로 둔다.
- 백필(Backfill)과 증분 수집(Incremental Collection)이 같은 데이터를 동시에 쓰면 DB 유니크 제약(Unique Constraint)과 upsert로 충돌을 허용하고, 작업 우선순위는 증분 수집을 우선한다.
- API 호출 제한(rate limit)은 M1에서 수집 워커 내부 전역 rate limiter로 관리하고, 후속 다중 워커에서는 분산 rate limiter를 결정한다.
- 업비트 API 정상 응답 원문(raw response)은 기본 저장하지 않고, 오류/스키마 불일치/파싱 실패/샘플만 제한 저장한다.
- 금액, 수량, 거래대금, 스프레드(Spread), 누적 잔량, 등락률, 호가 불균형(Imbalance)은 DB에서 `numeric`, Python에서 `Decimal`로 다룬다.
- API 응답에서 Decimal 값은 문자열로 보낸다.
- M1 코인 상세 화면은 읽기 전용 캔들(Candle) 차트와 호가 요약(Orderbook Summary) 패널로 제한한다.
- 기술적 분석 지표는 MVP 이후 필수 기능으로 명시한다.
- 파생 캔들(Derived Candle)은 운영 서버가 DB 원천 캔들(Source Candle)을 조회해 계산한다.
- 파생 캔들은 `complete`, `partial`, `empty` 완전성 상태를 함께 반환한다.
- 백필 계획(Backfill Plan)의 예상 요청 수와 저장 예상량은 보수적 추정치로 계산한다.
- M1 데이터 보존 정책은 삭제 없음으로 고정한다.
- 후속 M3 또는 M3.5에서 보존 기간, 파티셔닝(Partitioning), 압축, 다운샘플링(Downsampling), 삭제 정책, 삭제 후 재수집(Destructive Rebuild), 감사(Audit) 정책을 반드시 결정한다.
- M1 완료 기준은 제품 세로 절편 기준으로 정의한다. 백엔드 수집만 끝난 상태나 빈 화면만 있는 상태는 M1 완료로 보지 않는다.
- M1 완료에는 Docker Compose 실행, 후보 유니버스 갱신, 기본 활성 수집 대상 50개 설정, 주요 데이터 수집/저장, 운영 대시보드, 수집 대상/설정 화면, 백필 제어, 시장 리스트, 코인 상세, 계약/통합/API/브라우저 E2E(End-to-End) 테스트 통과가 포함된다.
