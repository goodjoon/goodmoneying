## 목표

백필 수집 워커(Backfill Collection Worker)가 이미 저장된 캔들 구간을 업비트에 다시 요청하지 않고, 없는 결측 구간(Missing Range)만 fetch하며, DB 저장은 기본 최대 3000개 batch 단위로 upsert하도록 구현한다. fetch 성공 시 heartbeat를 갱신하고, `rows_written`과 `last_completed_at`은 DB batch upsert 완료 후에만 갱신한다. 실패(failed)한 백필 작업은 Backfill 관리 화면에서 재개할 수 있고, 재개 시에도 이미 저장된 데이터는 업비트에 요청하지 않는다. 워커 로그 레벨은 운영 설정으로 변경할 수 있어야 한다.

## 현재 맥락

현재 구현은 `_missing_source_candle_ranges()`로 저장된 분(minute)을 제외한 결측 범위를 계산하지만, `fetch_minute_candles()`가 결측 구간 전체 row를 리스트로 반환한 뒤 `record_backfill_candles()`가 한 번에 upsert한다. 긴 결측 구간에서는 fetch 반복 중 heartbeat와 진행 상태 갱신이 늦어진다.

## 설계 메모

- fetch page 크기는 업비트 API 기준 200개로 유지한다.
- DB batch 크기는 fetch page와 분리하고 기본 최대 3000개 row로 둔다.
- batch 크기 설정은 1 이상의 정수만 허용한다.
- `on_progress` callback은 fetch 성공 직후에도 호출해 heartbeat를 갱신한다.
- `rows_written_count`는 DB upsert 반환값 누적을 기준으로 한다.
- `last_completed_at`은 batch에 실제 포함되어 DB upsert가 끝난 캔들의 최대 `candle_start_at`을 기준으로 한다. 빈 batch는 완료 시각을 전진시키지 않는다.
- 구현은 기존 API 계약의 `currentTargetBackfillRowCount`, `processedMissingRangeCount`, `estimatedMissingRangeCount` 표시를 깨지 않는 additive 내부 변경으로 처리한다.

## 실패 케이스

- 기존 데이터가 있는 구간까지 업비트에 재요청해 API quota와 실행 시간이 낭비된다.
- fetch는 성공했지만 heartbeat가 갱신되지 않아 worker가 `지연`으로 표시된다.
- DB batch upsert 실패 전에 `rows_written` 또는 `last_completed_at`이 앞서 갱신되어 화면 진행률이 실제 저장 상태보다 앞선다.
- pause/stop 요청 후 batch 경계에서 다음 target 또는 다음 job 처리 정책이 깨진다.
- `GOODMONEYING_BACKFILL_BATCH_SIZE=0`, 음수, 비정수 값이 조용히 허용된다.
- failed 작업 카드에 재개 버튼이 없어 운영자가 수동 안전 재시작만 선택하게 된다.
- failed 작업 재개가 저장 상태를 다시 계산하지 않아 이미 저장된 데이터를 업비트에 재요청한다.
- 운영 배포 후 로그 레벨을 조정할 수 없어 장애 시 job, target, fetch, DB batch 경계를 추적하기 어렵다.
