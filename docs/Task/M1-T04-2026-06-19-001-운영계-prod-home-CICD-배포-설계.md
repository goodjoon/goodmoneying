## 범위

- GitHub Actions organization self-hosted runner는 Mac Mini M4에 있는 기존 runner를 사용한다.
- 배포 트리거(trigger)는 `release` 브랜치 push로 한다.
- 이번 자동 배포 대상은 `prod-home` 프로필(profile) 하나로 제한한다.
- private GHCR(GitHub Container Registry)에 `api`, `worker`, `web` 이미지를 push한다.
- `prod-home` 서버 역할은 아래와 같이 고정한다.

| 서버 | 배포 역할 | 실행 서비스 |
|---|---|---|
| Mac Mini M4 | infra 서버, 배포 제어 노드 | `postgres`, GitHub Actions runner |
| APP SERVER 01 | application 서버 | `api`, `worker` |
| bmax-ubuntu | web 서버 | `web` |

- 운영 접근은 Tailscale 내부망 전용으로 한다.
- 배포 스크립트는 배포 프로필을 입력받는 구조로 만들되, 첫 구현은 `prod-home`만 허용한다.
- Slack 배포 명령은 후속 범위로 분리한다. 단, 수동 실행(`workflow_dispatch`) 진입점은 향후 Slack 봇이 호출할 수 있는 형태로 남긴다.

## 현재 맥락

현재 저장소는 `apps/api`, `apps/worker`, `apps/web`, `packages/shared` 구조를 사용한다. `docker-compose.yml`은 로컬 개발 기준으로 `postgres`, `api`, `worker`, `web`을 함께 실행한다. 각 앱에는 Dockerfile이 있으며, Python은 `uv`, 프론트엔드(Frontend)는 `npm`과 Vite 기반이다.

프로젝트 아키텍처는 M1에서 단일 수집 워커(Collection Worker), 단일 운영 서버(Operations Server), PostgreSQL 단일 저장소를 선택했다. M3.5 전까지 다중 워커와 고가용성은 도입하지 않는다. 이번 CI/CD 설계는 해당 결정을 유지하면서 운영계(prod) 배포 재현성과 검증 기준을 만든다.

장비 문서 기준 홈 인프라에는 `bmax-ubuntu`, `APP SERVER 01`, `Mac Mini M4`가 있고 모두 Tailscale로 접근 가능하다. `bmax-ubuntu`는 CPU/RAM 여유가 작으므로 web 정적 UI 실행만 맡기고, `APP SERVER 01`이 `api`와 `worker`를 담당한다. `Mac Mini M4`는 PostgreSQL과 배포 제어 역할을 맡는다.

## 설계 메모

배포 대상은 배포 환경(Deployment Environment), 배포 인프라(Deployment Infrastructure), 배포 프로필(Deployment Profile)로 분리한다.

| 축 | 이번 값 | 의미 |
|---|---|---|
| 배포 환경(Deployment Environment) | `prod` | 운영계, 실제 운영 데이터 사용 |
| 배포 인프라(Deployment Infrastructure) | `home` | 집에 있는 서버 묶음 |
| 배포 프로필(Deployment Profile) | `prod-home` | `prod`와 `home`을 묶은 실제 배포 대상 |

예상 파일 구조는 아래와 같다.

```text
deploy/
  profiles/
    prod-home/
      profile.env
      hosts.env
      compose.infra.yml
      compose.app.yml
      compose.web.yml
      README.md
  scripts/
    deploy-profile.sh
    healthcheck-profile.sh
```

GitHub Actions는 두 workflow로 나눈다.

| workflow | trigger | 책임 |
|---|---|---|
| `ci.yml` | PR, 모든 브랜치 push | 테스트, 린트(lint), 타입검사(type check), 빌드 검증 |
| `deploy.yml` | `release` 브랜치 push, 수동 실행(`workflow_dispatch`) | `prod-home` 이미지 push, 서버 배포, healthcheck, E2E(End-to-End) 검증 |

`release` push는 항상 `prod-home`으로 자동 배포한다. 수동 실행은 첫 구현에서 `prod-home`만 선택 가능하게 두고, 향후 `staging-home`, `prod-aws` 같은 배포 프로필이 생기면 선택지만 추가한다.

이미지 태그(tag)는 `release-{short-sha}`를 기본으로 한다. 필요하면 `prod-home-current` 같은 이동 태그를 추가할 수 있지만, 롤백 기준은 불변 태그인 `release-{short-sha}`로 둔다.

배포 단계는 아래 순서를 따른다.

1. 소스 체크아웃(checkout)
2. Python 검증: `uv run ruff check .`, `uv run mypy apps/api apps/worker packages/shared tests`, `uv run pytest`
3. Web 검증: `npm ci`, `npm test`, `npm run build`
4. fixture/mock 기반 E2E 검증
5. Docker 이미지 빌드: `api`, `worker`, `web`
6. private GHCR 로그인과 이미지 push
7. `deploy/scripts/deploy-profile.sh prod-home release-{short-sha}` 실행
8. 서버별 compose pull/up
9. 배포 후 healthcheck
10. Tailscale 내부 URL 기준 운영 화면 E2E 검증

배포 동시 실행은 GitHub Actions `concurrency`를 `deploy-prod-home`으로 두어 막는다.

## 계약 변경

이번 설계 자체는 DB/API/message 계약을 변경하지 않는다. 다만 구현 시 운영 배포용 healthcheck endpoint, 운영 화면 base URL, worker 실행 모드가 현재 계약과 충돌하는지 확인해야 한다.

특히 현재 `apps/worker/Dockerfile` 기본 명령은 `--once` 실행이다. 운영 배포에서는 지속 실행 모드 또는 재시작 정책을 명확히 정의해야 한다.

## 실패 케이스

- GitHub Actions runner label이 맞지 않아 `deploy.yml`이 Mac Mini M4 runner에서 실행되지 않는다.
- private GHCR push 또는 운영 서버 pull 인증이 실패한다.
- `APP SERVER 01`이 Mac Mini M4 PostgreSQL에 Tailscale 내부망으로 접근하지 못한다.
- `bmax-ubuntu`의 web이 `APP SERVER 01` API base URL을 잘못 바라본다.
- healthcheck 실패 후 이전 버전으로 롤백하지 못한다.
- worker가 `--once`로 종료되어 실제 운영 수집이 지속되지 않는다.
- DB schema 변경 배포가 자동 적용되어 운영 데이터 손상 위험을 만든다.

## 완료 기준

- `release` 브랜치 push가 `prod-home` 배포 workflow를 자동 실행한다.
- `main`과 PR은 배포 없이 CI 검증만 수행한다.
- `api`, `worker`, `web` 이미지가 private GHCR에 `release-{short-sha}` 태그로 push된다.
- `Mac Mini M4`, `APP SERVER 01`, `bmax-ubuntu`에 서버 역할별 compose 배포가 수행된다.
- 배포 후 API `/health`, web HTTP 200, PostgreSQL healthcheck, worker 실행 상태 검증이 통과한다.
- Tailscale 내부 URL 기준 E2E 테스트가 통과한다.
- 배포 실패 시 이전 `release-{short-sha}` 태그로 롤백할 수 있다.
