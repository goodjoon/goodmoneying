## 현재 맥락

구현 시작 시점의 저장소에는 M1 제품 기준, 아키텍처, DB/API 계약, ADR만 있고 애플리케이션 코드와 실행 설정은 없었다. 구현 후에는 `uv`, `npm`, `pytest`, `Vitest`, `Playwright` 검증뿐 아니라 standalone `docker-compose`와 Podman 소켓(socket)을 통한 PostgreSQL/API/worker/web 컨테이너 실행도 확인했다.

## 설계 메모

### 접근안

| 접근 | 장점 | 단점 | 판단 |
|---|---|---|---|
| 한 번에 풀스택 구현 | 왕복이 적음 | 실패 지점이 커지고 계약 drift를 늦게 발견함 | 사용하지 않음 |
| 백엔드 우선 후 화면 | 데이터 경계가 먼저 안정됨 | 브라우저 E2E와 화면 요구사항이 뒤로 밀림 | 일부만 사용 |
| 계약 검증 스캐폴드부터 세로 절편을 점진 확장 | DB/API/워커/화면을 작은 검증 단위로 닫음 | 초기 계획 문서가 길어짐 | 채택 |

### `grill-me` 결정 기록

- 질문: 구현 전 기술 스택을 다시 고를 것인가?
  - 추천 답: 다시 고르지 않는다. `ADR-0001`이 Accepted이므로 Python, FastAPI, React, PostgreSQL, Docker Compose를 기준으로 구현한다.
- 질문: PostgreSQL이 로컬에 없을 때 테스트 DB를 어떻게 둘 것인가?
  - 추천 답: 런타임 계약은 PostgreSQL로 유지한다. 단위/통합 테스트는 저장소 포트(Repository Port)를 통해 SQLite 호환 테스트 어댑터를 사용하고, PostgreSQL DDL 파싱/적용 검증은 Docker 또는 psql 가능 환경에서 별도 실행할 수 있게 분리한다.
- 질문: 실제 업비트 API 호출을 기본 테스트에 포함할 것인가?
  - 추천 답: 기본 자동화 테스트는 fixture/mock 기반으로 고정하고, 실제 API 호출은 `live` 프로필로 분리한다.
- 질문: M1 완료를 백엔드 완료로 볼 것인가?
  - 추천 답: 보지 않는다. 운영 대시보드, 수집 대상/설정, 백필 작업, 시장 리스트, 코인 상세 화면과 브라우저 E2E까지 통과해야 한다.

## 실패 케이스

- 업비트 API 응답 지연 또는 장애
- fixture와 실제 API 응답의 필드 drift
- 운영 토큰 누락 또는 오입력
- 활성 수집 대상이 정확히 50개가 아닌 요청
- 후보 유니버스 밖 거래 상품이 활성 수집 대상으로 들어오는 요청
- 백필 작업 제어 중 잘못된 상태 전이
- 현재가/호가/캔들 최신성 기준을 만족하지 못하는 지연 상태
- 컨테이너 런타임(Container Runtime) 미설치 또는 소켓(socket) 미연결로 인한 실행 검증 공백
